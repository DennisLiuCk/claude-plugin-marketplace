## 概述

**Skill Creator** 是 Anthropic 官方推出的一項「用 AI 建立 AI 技能」的元技能（meta-skill）。使用者透過 `/skill-creator` 指令，即可在 Claude 的引導下，從零開始建立、測試和優化一項 Claude 技能。

2026 年 3 月 3 日，Anthropic 發佈了 Skill Creator 的重大更新。社群對新版本的反應非常正面，普遍認為效果有了顯著提升。本文檔將深入分析新舊版本的差異，並拆解官方使用了哪些策略讓這次升級如此有效。

### 為什麼這份分析很重要？

Skill Creator 是一個「技能教你寫技能」的工具——它本身就是 SKILL.md 最佳實踐的展示。分析它的演進過程，等於是在學習 Anthropic 內部團隊如何設計高品質的 AI 驅動工作流程。

## 檔案架構比較

### 舊版結構（2025 年 10 月）

```
skill-creator/
└── SKILL.md              # 單一檔案，包含所有指引
```

舊版的 Skill Creator 本質上是一份「教學文檔」。整份 SKILL.md 讀起來更像是給人類看的開發者文件，而非給 Claude 執行的操作指令。

### 新版結構（2026 年 3 月）

```
skill-creator/
├── SKILL.md                    # 核心指引（精簡後的決策框架）
├── LICENSE.txt                 # 授權檔案
├── agents/                     # 專職代理定義
│   ├── analyzer.md             # 分析代理：識別模式、建議改進
│   ├── comparator.md           # 比較代理：盲測 A/B 比較
│   └── grader.md               # 評分代理：量化斷言評估
├── scripts/                    # 自動化腳本（9 個）
│   ├── __init__.py
│   ├── aggregate_benchmark.py  # 彙整基準測試結果
│   ├── generate_report.py      # 產生報告
│   ├── improve_description.py  # 優化觸發描述
│   ├── package_skill.py        # 打包技能以供分發
│   ├── quick_validate.py       # 快速驗證技能結構
│   ├── run_eval.py             # 執行評估測試
│   ├── run_loop.py             # 迭代優化迴圈
│   └── utils.py                # 共用工具函數
├── eval-viewer/                # 評估結果檢視器
│   └── (HTML 模板與靜態資源)
├── references/                 # 參考資料
│   └── schemas.md              # JSON 結構定義文件
└── assets/                     # 靜態資源
    └── eval_review.html        # 觸發評估審核介面
```

### 架構差異的核心意義

| 面向 | 舊版 | 新版 |
|------|------|------|
| 檔案數量 | 1 個 | 20+ 個 |
| 組織方式 | 單一文檔 | 模組化分工 |
| 執行能力 | 純指引 | 指引 + 可執行腳本 |
| 代理數量 | 0 | 3 個專職代理 |
| 自動化程度 | 無 | 高度自動化 |

這個變化反映了一個重要的設計哲學轉變：**從「告訴 Claude 怎麼做」到「給 Claude 工具讓它自己做」**。

## 運作模式比較

### 舊版：單一 Create 模式

```
使用者意圖 → Claude 訪談 → 撰寫 SKILL.md → 手動測試 → 完成
```

舊版的工作流程是線性的：
1. 使用者描述想要的技能
2. Claude 提問以釐清需求
3. Claude 產生 SKILL.md
4. 使用者自行測試（無自動化支援）
5. 如果不滿意，手動迭代

### 新版：四種可組合模式

#### 1. Create 模式（建立）
```
使用者意圖 → 訪談與研究 → 撰寫 SKILL.md → 建立 evals.json → 自動測試 → 迭代
```
改進重點：建立技能的同時就設定測試案例，而非事後補充。

#### 2. Eval 模式（評估）
```
載入 evals.json → 平行生成子代理 → 同時執行有技能/無技能的測試 → 評分 → 產生報告
```
核心創新：以**平行多代理**方式同時運行測試，每個測試案例都有獨立上下文。

#### 3. Improve 模式（改進）
```
載入回饋 → 分析失敗模式 → 泛化改進（非針對特定案例）→ 重新測試 → 迴圈
```
核心原則：改進必須**泛化**——如果技能只對測試案例有效，就是無用的。

#### 4. Benchmark 模式（基準測試）
```
分割評估集(60/40) → 評估當前版本 → AI 提出改進 → 迭代最多 5 次 → 選出最佳版本
```
核心機制：使用**留出測試集**防止過度擬合，自動選擇泛化能力最好的版本。

### 模式組合的威力

這四種模式並非孤立運作。一個典型的完整流程是：

```
Create → Eval → Improve → Eval → Improve → ... → Benchmark → 完成
```

每次 Improve 後都可以跑 Eval 來確認改進是否有效，最後用 Benchmark 做全面驗證。這形成了一個**可量化的迭代迴圈**。

## 六大關鍵升級策略

### 策略一：多代理協作架構

這是新版最核心的架構變化。官方引入了四個可組合的專職代理：

| 代理 | 職責 | 設計用意 |
|------|------|----------|
| **Executor** | 在隔離環境中執行技能 | 確保每次測試都在乾淨的上下文中進行 |
| **Grader** | 根據預定義斷言評分 | 標準化的輸出格式：`text`、`passed`、`evidence` |
| **Comparator** | 盲測比較兩個版本 | 消除知道「哪個是新版」的認知偏見 |
| **Analyzer** | 分析結果模式並建議改進 | 從失敗案例中提取可泛化的改進方向 |

**為什麼這很重要？**

舊版中，所有工作都由同一個 Claude 對話負責。這會導致：
- 上下文混淆：測試結果與技能內容混在一起
- 偏見累積：Claude 記得自己寫了什麼，評估時不夠客觀
- 無法平行化：只能逐一測試

新版透過代理分工，每個代理有自己的上下文視窗、專用工具清單和明確職責，徹底解決了這些問題。

**技術要點：** 每個代理定義在獨立的 `.md` 檔案中（`agents/` 目錄），包含 YAML 前置資料指定模型、工具和觸發條件。這實踐了 SKILL.md 文件中推薦的「漸進式揭露」原則。

### 策略二：自動化評估系統（Eval）

新版引入了一套完整的測試基礎設施：

```json
// evals/evals.json 範例結構
[
  {
    "prompt": "幫我建立一個用於程式碼審查的技能",
    "expected": "應該產生包含 SKILL.md 的技能資料夾結構",
    "assertions": [
      {
        "text": "輸出包含 YAML 前置資料",
        "type": "quantitative"
      }
    ]
  }
]
```

**執行流程：**
1. 為每個測試案例同時啟動兩個子代理：一個載入技能，一個作為基線
2. **在測試執行的同時**草擬量化斷言（不是事後，而是並行）
3. 測試完成後立即捕獲效能資料（`total_tokens`、`duration_ms`）
4. 使用 Grader 代理評估所有斷言
5. 透過 eval-viewer 產生 HTML 報告供人工審核

**設計巧妙之處：** 斷言在測試「執行中」而非「完成後」草擬。這避免了「看到結果才決定標準」的事後偏見。

### 策略三：盲測 A/B 比較機制

這是從學術研究方法論借鑑的嚴謹做法：

```
版本 A 輸出 ──┐
              ├──→ Comparator 代理（不知道哪個是 A/B）──→ 判定勝者
版本 B 輸出 ──┘
```

Comparator 代理在評估時：
- **不知道**哪個輸出來自新版、哪個來自舊版
- 僅根據輸出品質進行獨立判斷
- 判定完成後才揭曉身份並分析原因

**為什麼這比直接比較更好？**

當你告訴 AI「這是改進後的版本」，它會傾向給出正面評價——這是確認偏誤。盲測完全消除了這個因素。

### 策略四：描述觸發優化器

Anthropic 發現，技能觸發的準確度是決定使用體驗的關鍵。Claude 天生有「低觸發」（undertriggering）的傾向——即使技能適用，也可能不主動使用。

新版的優化流程：

1. **產生評估查詢**：自動生成 20 個查詢（混合「應觸發」和「不應觸發」的情境）
2. **人工審核**：透過 HTML 模板讓使用者調整評估集
3. **自動迭代優化**：
   - 將評估集分割為 60% 訓練集 / 40% 測試集
   - 用訓練集優化描述
   - 用測試集驗證泛化能力
   - 最多迭代 5 次
   - 選出**測試集分數最高**的版本（非訓練集）
4. **套用最佳描述**：更新 SKILL.md 前置資料

**防止過度擬合的關鍵：** 以留出測試集的分數決定最佳版本，而非訓練集。這確保優化後的描述在未見過的查詢上也能表現良好。

在官方測試中，此優化器對 6 個公開文件建立類技能進行測試，**其中 5 個的觸發準確度獲得了改善**。

### 策略五：漸進式揭露（Progressive Disclosure）

新版 Skill Creator 將自身作為漸進式揭露的最佳實踐範例：

```
第一層：元資料（name + description）     ← 永遠載入，用於觸發判斷
第二層：SKILL.md 本體（≤500 行）         ← 觸發後載入，提供核心指引
第三層：捆綁資源（references/、scripts/） ← 按需載入，提供深入支援
```

舊版的問題在於把所有內容塞進單一 SKILL.md，接近 500 行上限。這不僅浪費 token，也讓 Claude 難以快速定位關鍵資訊。

新版將詳細的 schema 定義移到 `references/schemas.md`，將自動化邏輯移到 `scripts/`，讓 SKILL.md 專注於**決策框架**和**執行流程**。

**這對你的技能開發意味著什麼？**
- 核心指引保持精簡
- 參考資料按需載入
- 腳本負責可自動化的工作
- 每一層都有明確的職責

### 策略六：反過度擬合策略

新版 SKILL.md 中反覆強調一個核心理念：

> 「如果你和使用者共同開發的技能只對那些範例有效，那它就是無用的。」

這體現在多個層面：

1. **Improve 模式的改進原則**：
   - 從回饋中**泛化**模式，而非針對特定案例修補
   - 保持提示精簡，移除沒有實際效果的元素
   - 解釋推理過程，讓模型理解指令背後的「為什麼」

2. **Benchmark 模式的驗證機制**：
   - 使用留出測試集防止對訓練集過度擬合
   - 每個查詢執行 3 次取平均，減少單次隨機性
   - 選出泛化能力最好的版本，而非訓練分數最高的版本

3. **描述優化的分割策略**：
   - 60/40 的訓練/測試分割
   - 最終以測試集分數為準

## 實際案例：PDF 技能的改進

Anthropic 在官方部落格中分享了一個實際案例，展示 Eval 模式如何幫助發現並修復問題：

**問題：** PDF 技能在處理非可填寫表單時失敗。Claude 需要在沒有已定義欄位的位置精確放置文字。

**舊版做法：** 依靠使用者手動發現問題並回報。

**新版做法：**
1. 透過 Eval 模式建立包含非可填寫表單的測試案例
2. 自動化測試精確定位了失敗的環節
3. 修復方案：將文字定位錨定到提取的文字座標
4. 再次執行 Eval 確認修復有效

這個案例完美展示了「測試驅動改進」的威力——問題被系統性地發現、隔離、修復和驗證。

## 對技能開發者的啟示

### 1. 建立技能時就考慮可測試性

不要等技能完成後才想「怎麼測試」。在設計階段就定義：
- 什麼輸入應該觸發這個技能？
- 什麼輸入不應該觸發？
- 好的輸出長什麼樣？
- 可以用什麼斷言來量化品質？

### 2. 善用代理分工

如果你的技能涉及複雜的工作流程，考慮將不同職責拆分為獨立代理：
- 每個代理有明確的單一職責
- 代理之間透過結構化資料溝通
- 避免在同一上下文中累積過多無關資訊

### 3. 實踐漸進式揭露

將技能內容分層：
- **SKILL.md**：只放決策框架和執行步驟
- **references/**：放詳細的參考資料和 schema
- **scripts/**：放可自動化的工作
- 總行數控制在 500 行以內

### 4. 描述是觸發的關鍵

技能描述不僅是給人看的說明——它是 Claude 決定是否使用這個技能的唯一依據。官方建議：
- 使用稍微「強勢」的措辭（例如「務必在...時使用此技能」）
- 同時包含「做什麼」和「什麼時候用」的資訊
- 對抗 Claude 天生的低觸發傾向

### 5. 迭代時泛化，而非特化

每次改進技能時，問自己：
- 這個修改是否只對某個特定案例有效？
- 改進的邏輯能否推廣到類似情境？
- 有沒有移除無效的指令，而不只是新增？

### 6. 用資料驅動決策

引入量化指標來判斷改進是否有效：
- token 使用量是否合理？
- 執行時間是否在預期範圍？
- 與基線相比，通過率是否提升？
